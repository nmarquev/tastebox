```bash
npm run dev               # Vite dev server on port 8080
npm run build             # Production build
npm run lint              # ESLint
```

```bash
npm run dev               # tsx watch server on port 3001 (HTTPS)
npm run build             # Compile TypeScript
npm run db:migrate        # Create & run Prisma migration
npm run db:push           # Push schema without migration (dev only)
npm run db:generate       # Regenerate Prisma client after schema changes
npm run db:seed           # Seed test data
```

## Architecture

**Monorepo**: React frontend (Vite) + Express backend + Chrome Extension (Manifest V3).

### Frontend (`src/`)
- **Entry**: `src/main.tsx` → `App.tsx` (providers stack: ErrorBoundary > QueryClient > ThemeProvider > AuthProvider)
- **Single page**: `src/pages/Index.tsx` — recipe grid with filtering, modal management, TTS
- **State**: React Context (`AuthContext`, `ThemeContext`) + TanStack React Query for server state
- **API client**: `src/services/api.ts` — all backend calls go through here
- **Types**: `src/types/recipe.ts` — shared Recipe, Ingredient, Instruction interfaces
- **Path alias**: `@/` maps to `src/`

### Backend (`backend/src/`)
- **Entry**: `index.ts` — Express + optional HTTPS (self-signed certs), Helmet, CORS
- **Auth**: JWT via `middleware/auth.ts`; tokens stored in localStorage as `auth_token`
- **Routes**: one file per feature in `routes/` (recipes, auth, importHtml, importDocx, importPdf, nutrition, upload, imageProxy, llm, pdf)
- **Services**: business logic in `services/` — `llmServiceImproved.ts` (OpenAI calls), `pdfProcessor.ts`, `docxProcessor.ts`, `imageService.ts` (Sharp optimization), `cookidooService.ts`
- **Validation**: Zod schemas at the route level; use `.nullable().optional()` pattern (not `.optional().nullable()`)

### Database (Prisma + SQLite dev / PostgreSQL prod)
Schema at `backend/prisma/schema.prisma`. Key models:
- **Recipe** — includes nutritional fields (calories, protein, carbs, fat, fiber, sugar, sodium)
- **Ingredient** — has `section` field for multi-part recipes
- **Instruction** — has `function`, `time`, `temperature`, `speed`, `section` (Thermomix-specific)
- **RecipeTag** — junction table

After any schema change: `npm run db:generate` then `npm run db:migrate`.

### Chrome Extension (`extension/`)
Manifest V3. `js/config.js` controls dev vs production API URLs (`isDevelopment` flag). Background service worker (`background.js`) handles auth token persistence in Chrome storage and API calls. Content script (`content.js`) extracts page HTML.

## Key Patterns

**Import flows**: All recipe imports (URL, HTML, PDF, DOCX, Chrome Extension) funnel through `llmServiceImproved.ts` which calls OpenAI. Each import type has its own route in `backend/src/routes/`.

**Images**: Downloaded, optimized via Sharp, stored in `backend/uploads/`. Served via Express static + `imageProxy` route for CORS bypass.

**TTS**: `Web Speech API` in browser; narrative script generated by OpenAI. Settings persisted in `localStorage` via `useVoiceSettings` hook.

**Nutrition**: Auto-calculated via OpenAI from ingredient list; stored directly on the Recipe model.

## Environment Variables

Backend needs `backend/.env`:
```
DATABASE_URL=file:./dev.db
OPENAI_API_KEY=sk-...
JWT_SECRET=...
PORT=3001
FRONTEND_URL=http://localhost:8080
```

Frontend (optional) `VITE_API_URL` — defaults to `http://localhost:3001` in dev.

## Production

- Domain: `https://tastebox.beweb.com.ar`, backend on port 5000
- PM2 via `ecosystem.config.cjs`, Nginx handles SSL termination
- DB path: `./db/tastebox.db` (see `.env.production.example`)
- Extension ZIP: run `npm run package-extension`, upload to `/downloads/` on server
